DocSeek — Acceptance test 6 (manual)
User story: As a patient looking for something specific, I want to filter my recommendations by my location and availability, so that I can only see doctors tailored to my preferences.

---

Local setup — run DocSeek on your machine (not deployed)

This test needs the **client**, **API**, **Postgres**, **OpenAI** (symptom validation and doctor search), and **loaded doctor data** so results and filters behave realistically.

### A. What to install first

1. **Git** — to clone the repository.
2. **Docker Desktop** (or Docker Engine with Compose v2) — runs Postgres (pgvector), the Bun API, and the Vite client. Start Docker and wait until it is fully running.
3. **OpenAI API access** — the API uses OpenAI for validation and search. You need an **`OPENAI_API_KEY`** in `.env` (billing may apply per OpenAI).
4. **uv** — documented way to load sample doctors. Install from https://docs.astral.sh/uv/getting-started/installation/

### B. Get the code

1. Clone the DocSeek repository (or download and unzip it).
2. Open a terminal and **cd** into the **repository root** (folder containing `client/`, `api/`, `data-scripts/`, and `compose.yml`).

### C. Create environment configuration

1. In the **repository root**, copy the example file:

   ```bash
   cp .env.example .env
   ```

2. Edit `.env` in a text editor.

3. Set **`OPENAI_API_KEY=`** to your real key. The API **will not start** without it; it reads the repo root `.env` when the project is mounted into containers.

4. For **host-run scripts** (repopulate on your machine), keep **`DATABASE_URL`** as in `.env.example`, typically:

   `postgresql://docseek:docseek@localhost:55432/docseek_upmc`

   That matches Postgres published on host port **55432** by Compose. The API container uses its own internal `DATABASE_URL` to reach the `postgres` service.

5. Keep **`CORS_ALLOWED_ORIGINS`** including `http://localhost:5173` and `http://127.0.0.1:5173`.

6. Save `.env`.

### D. Start Postgres, API, and client

1. From the **repository root**:

   ```bash
   docker compose up
   ```

   If needed:

   ```bash
   docker compose -f compose.yml up
   ```

2. First launch may take several minutes (image pull, `bun install` in `api` and `client`).

3. When healthy:

   - **Web app:** http://localhost:5173  
   - **API:** http://localhost:3000  

4. **Collation / old Postgres volume:** if you see a collation mismatch warning, reset once (root `README.md`):

   ```bash
   docker compose down -v
   rm -rf data-scripts/.docker-postgres
   docker compose up
   ```

### E. Load doctor data (required for this test)

You need doctors in the database with **locations** and **accepting_new_patients** values so location and availability filters can do something visible.

1. On your **host**, with Compose running, open a second terminal:

   ```bash
   cd data-scripts
   uv run python repopulate_database.py
   ```

2. Wait until it completes without errors (loads `upmc_doctors.json`, refreshes embeddings; requires network access to OpenAI).

### F. Verify before you test

1. Open **http://localhost:5173** and confirm the home page loads (“How can we help you today?”).
2. Run a quick search **without** filters: you should get **Recommended doctors** after **Loading recommendations…**
3. If search fails, check API logs and `.env` `OPENAI_API_KEY`.

---

Prerequisites specific to this test

1. **Desktop browser** with JavaScript enabled.
2. **Geolocation (optional):** if the browser asks for location on the results page, **Allow** if you want real distances next to addresses; **Deny** is OK—the app may show a fallback distance. Filtering still works.

---

Steps
1. Open **http://localhost:5173**.
2. In the main symptom field, describe what you want care for (e.g. a suggested chip or your own text). Add detail until validation lets you continue.
3. Scroll to **Filter by your preferences.**
4. In **Location (city, state, or ZIP),** enter an area that plausibly matches your dataset (placeholder example: **Pittsburgh, PA**).
5. Check **Only show doctors accepting new patients** if you want that constraint; leave unchecked for no availability filter.
6. Submit with the **round arrow** on the home search bar (screen readers: **Find matching doctors**) so you land on **Recommended doctors.**
7. Confirm the results page shows your symptoms and, when filters were used, **Filtered by:** your location and/or **Accepting new patients**, plus **Refine filters.**
8. On the first card, read the practice **location** line; if you allowed geolocation, note **distance**; otherwise note fallback distance behavior.
9. Read the availability line (**Accepting new patients** vs. **Check availability**) and confirm it matches the checkbox when that filter was on.
10. Click **See the next recommended doctor** at least once; each card should still match the same filtered set.
11. Click **Refine filters.** Change location and/or **Only show doctors accepting new patients**, then **Apply filters.**
12. Confirm **Filtered by** (and/or the URL) updates, the list reloads, and the first recommendation matches the new filters—or you see the empty-state message if nothing matches.
13. Optional: **Start a new search**, return home, repeat steps 2–6 with different filters.

Expected result
You can constrain recommendations by **location** and **accepting new patients** before and after the first search, see those constraints on the results page, and browse a filtered list (or a clear no-match message).

---

Optional: run without Docker (advanced)

With Postgres on **localhost:55432** matching `.env.example`, install **Bun**, run `bun install` and `bun run dev` in `api/` and `client/`, set **`VITE_API_BASE_URL=http://localhost:3000`**, and keep **`OPENAI_API_KEY`** in the repo root `.env`. Prefer Docker Compose unless you use a native workflow.
