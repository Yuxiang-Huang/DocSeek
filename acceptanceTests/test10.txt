DocSeek — Acceptance test 10 (manual)
User story: As a patient exploring options, I want to save physicians I am interested in so that I can revisit them later without repeating my search.

What this feature does
Saved doctors are stored in your **browser** (local storage) on this device. The top nav shows **Saved physicians** and a **count** when you have saves. The **Saved physicians** page lists everyone you saved so you can open profiles or booking links again without running the symptom search again.

---

Local setup — run DocSeek on your machine (not deployed)

This project is a small monorepo: a React client (`client/`), a Bun API (`api/`), and Postgres with pgvector. The usual way to run everything locally is **Docker Compose** from the repo root, then load sample doctor data once.

### A. What to install first

1. **Git** — to clone the repository.
2. **Docker Desktop** (or another Docker engine with Compose v2) — runs Postgres, the API, and the Vite dev server in containers.  
   - After installing, start Docker and wait until it reports “running.”
3. **OpenAI API access** — symptom validation and doctor search use OpenAI from the API. You need an API key from your OpenAI account (billing may apply per their pricing).
4. **uv** (Python package runner) — used to run the checked-in database repopulate script. Install from https://docs.astral.sh/uv/getting-started/installation/ (one-line installers are listed there).  
   - If you cannot use `uv`, you may run the same script with `python repopulate_database.py` from `data-scripts/` after installing its dependencies yourself; the repo expects `uv` as in the root README.

### B. Get the code

1. Clone the DocSeek repository to your computer (or download and unzip it).
2. Open a terminal and **change directory into the repository root** (the folder that contains `client/`, `api/`, `data-scripts/`, and `compose.yml`).

### C. Create environment configuration

1. In the **repository root**, copy the example env file to a real `.env` file:

   ```bash
   cp .env.example .env
   ```

2. Open `.env` in a text editor.

3. Set **`OPENAI_API_KEY=`** to your actual key (no quotes needed unless your editor adds them). The API **will not start** without this variable; it reads the repo root `.env` when the container mounts the project.

4. Leave **`DATABASE_URL`** as in `.env.example` **for scripts you run on your Mac/PC** (host machine):

   `postgresql://docseek:docseek@localhost:55432/docseek_upmc`

   That matches Postgres when it is published to port **55432** by Compose (see `compose.yml`). Inside Docker, the API service overrides `DATABASE_URL` to use the internal hostname `postgres`; you do not need to change that for Compose.

5. Keep **`CORS_ALLOWED_ORIGINS`** including `http://localhost:5173` and `http://127.0.0.1:5173` so the browser can call the API.

6. Save `.env`.

### D. Start Postgres, API, and client with Docker Compose

1. From the **repository root**, run:

   ```bash
   docker compose up
   ```

   (If your Docker version requires an explicit file: `docker compose -f compose.yml up`.)

2. The first run may take several minutes: images download, `bun install` runs in the API and client containers, and Postgres initializes.

3. When things look healthy, you should see logs suggesting the API is listening on port **3000** and Vite on **5173**.

4. Leave this terminal running, or start Compose in detached mode (`docker compose up -d`) and use `docker compose logs -f` to watch logs.

5. **If Postgres shows a collation version mismatch** (from an old data volume), stop everything and reset once as described in the root `README.md`:

   ```bash
   docker compose down -v
   rm -rf data-scripts/.docker-postgres
   docker compose up
   ```

### E. Load doctor data (so search returns results)

Saving physicians only matters after you can reach a **Recommended doctors** list. Load data from the checked-in JSON (do not scrape for a first run):

1. Open a **second terminal**; stay on your **host** (not inside a container).

2. Ensure Compose is up and Postgres is accepting connections on **localhost:55432**.

3. Run:

   ```bash
   cd data-scripts
   uv run python repopulate_database.py
   ```

4. Wait until the script finishes without errors. It reloads Postgres from `upmc_doctors.json` and refreshes embeddings (this needs network access to OpenAI for embedding generation).

### F. Verify the stack before the acceptance test

1. In a browser, open **http://localhost:5173** — you should see the DocSeek home page (“How can we help you today?”).

2. Optional sanity check: open **http://localhost:3000** only if your team exposes a health route; otherwise rely on the client loading without CORS errors (browser devtools → Network).

3. If the home page errors when searching, confirm the API container is running, `.env` has `OPENAI_API_KEY`, and repopulate completed.

---

Prerequisites specific to this test (browser)

1. Use a normal browser session (**not** a strict private/incognito mode that wipes storage every time) so **Save for later** persists as real users expect.
2. Use the **same browser and device** for “come back later” checks—saves are per-browser `localStorage`, not a cloud account.

---

Part A — Save from search results
1. From the home page, enter symptoms and run a search until you see **Recommended doctors** and a doctor card.
2. Find **Save for later** on the card (bookmark-style control).
3. Click **Save for later**. It should change to **Saved** (or show a saved state).
4. Look at the top navigation: **Saved physicians** should show a **number badge** when at least one doctor is saved.

Part B — Revisit without searching again
1. Click **Saved physicians** in the nav (accessibility label may include how many are saved).
2. Confirm you land on **Saved physicians** / **Your saved list** and see the doctor you saved.
3. Confirm you can read their details and use links such as **View profile** or **Book appointment** when present—without re-entering your symptoms on the home page.

Part C — Persistence (optional but recommended)
1. Refresh the page or close the tab, open **http://localhost:5173** again, and go straight to **Saved physicians**.
2. Confirm your saved doctor is still listed.

Part D — Clean up (optional)
1. On the saved card, use **Remove from saved** if you want to clear the test data.

Expected result
You can bookmark doctors from results, open **Saved physicians** later, and still see them without repeating the search—on the same browser and device.

---

Optional: run client and API without Docker (advanced)

If you already run Postgres on port **55432** with the same credentials/database name as `.env.example`, you can install **Bun** locally, run `bun install` and `bun run dev` in `api/` and `client/`, and set **`VITE_API_BASE_URL=http://localhost:3000`** for the client. This path is more error-prone for new testers; prefer Docker Compose unless you know you need a native workflow.
